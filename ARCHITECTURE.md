## 建议的分层结构

按**使用时刻**而非内容类型组织：

```
┌──────────────────────────────────────────────┐
│  L2 · AI 指令模板（重构方案）                  │
├──────────────────────────────────────────────┤
│                                               │
│  ▶ 阶段0：对齐闸（仅首次粘贴时使用）            │
│     - 让 AI 复述目标边界 + 完成理解 + 确认执行  │
│     - 一次性消耗品，审过后从上下文踢出          │
│                                               │
│  ▶ 阶段1：工作标准（全程锚定）                  │
│     - 铁律 + 完成定义（什么是"做完"）           │
│     - 分工规则（AI=知识源，人=审计）            │
│     - 谎报红线（确认 vs 疑似）                 │
│                                               │
│  ▶ 阶段2：动作清单（阶段输入时引用）            │
│     每项独立编号，带详细执行要求与绕过策略       │
│     ① 指纹→CVE（含 OAuth/SSO 专项补充）        │
│     ② JS 全量（含 CDN 绕过策略清单）            │
│     ③ FUZZ（注入/边界/污染/type confusion）     │
│     ④ 权限绕过（JWT/头/路径/方法）              │
│     ⑤ 攻击面穷举（业务逻辑层）                  │
│                                               │
│  ▶ 阶段3：生命周期 + 门禁条件（阶段流转用）      │
│     每个阶段只写：                             │
│       - 输入条件（上阶段的产物）                │
│       - 执行动作（引用动作清单的编号）           │
│       - 输出产物（符合产物硬要求）              │

│                                               │

│     - 每列禁止值（不许填"未做/N/A"）            │
│                                               │
│  ▶ 阶段5：追问扳手（AI 产出异常时补刀）          │

│     - 每次追问只问最关键的一句，不批量抛         │
│                                               │
│  ▶ 阶段6：完成度自检（AI 交付前自查清单）        │

│                                               │
```

## 关键设计决策

### 1. 为什么把"对齐闸"独立出来

当前 L2 开头是"粘贴这段"，AI 每次都被喂一遍铁律+完成定义+阶段说明。
对于长项目，AI 第一次跑完对齐后，这些内容占据无用上下文且不再被参考。

**方案**：对齐闸只出现在首次 prompt 中，AI 完成对齐确认后即从上下文移除。
后续每次执行只需引用"动作清单编号"和"当前阶段的产物硬要求"。

### 2. 阶段描述只做两件事

当前每个阶段内写了一大段话（阶段1 里写"穷举技术栈，执行强制①②查 CVE 啃 JS"）。
这在阶段流转时 AI 要重新 parse 整段，且和动作清单的定义重复。

**方案**：
```
阶段1 侦察
  输出：指纹表 | JS 资产表
  引用：动作①（指纹→CVE）、动作②（JS 全量）
  门禁：所有组件走到"版本识别 + CVE 比对结论"，
        所有 JS 走到"接口矩阵补全 + 敏感信息提取"
```

### 3. 产物硬要求固定为"不允许填 N/A"

这是一条**执行纪律而非信息**，不应该被当成阶段描述来写。
建议独立成一段放在生命周期描述之后，作为所有阶段的统一验收标准。

### 4. 追问扳手按主题分组

当前是扁平的 10 条列表。建议改为：

```
CVE 类：
  - "这个组件 CVE 搜了吗、验了吗？编号和结论？不能联网就声明。"
  - "CVE 编号可溯源吗？是只比版本号还是落了实测特征？"

JS 类：
  - "JS 全量啃了吗？大包跳过没？路由表解析了？"
  - "CDN 拦了 JS？绕过策略试了几个？CSS 引用路径看了吗？"

权限绕过类：
  - "JWT 四项（alg none/RS→HS/弱密钥/kid）都试了？"
  - "XFF/调试头/路径方法绕过都跑了？"

深度类：
  - "这只是识别，钻到利用或证伪了吗？证据呢？"
  - "业务逻辑层穷举了吗？还是只跑了技术指纹？"
```

这样人类审计时可以根据**当前卡在哪一步**，快速选对应的扳手补刀。

## 与 L1/L3 的边界

| 层 | 应该有的 | 不应该有的 |
|---|---|---|
| L2 | 可执行的动作指令、引用编号、绕过策略 | 原理推导、为什么这么做（属于 L1） |
| L1 | 原则、铁律、设计哲学 | 具体执行方法、绕过技术细节（属于 L2） |
| L3 | 实战中发现的漏报模式、新打法 | 未经验证的理论（放在本草案层） |

当前混淆点：L2 主指令块开头的"铁律""完成定义"其实是 L1 内容。
要不要在重构时彻底从 L2 中移除（改由对齐闸引用 L1），还是保留在 L2 开头作为 AI 的工作标准锚定？

## 待决策项

1. **对齐闸是否独立** — 独立后 L2 主指令块不再包含对齐内容，AI 首次和后续使用不同 prompt
2. **L1 原则是否保留在 L2 中** — 保留作"工作标准锚定" vs 完全引用 L1
3. **阶段描述的详细程度** — 仅写"输入/输出/门禁" vs 保留当前展开式描述
4. **产物硬要求的定位** — 统一放在生命周期后 vs 分散在各阶段下
5. **追问扳手的组织方式** — 扁平列表 vs 按主题分组
6. **本草案的存放位置** — 保留在 L2 底部作为注释，还是独立成文档，还是写进 AGENTS.md
