## 灵魂一句话

**真正的渗透是往下钻、钻到打穿；不是横着铺、铺到好看。**

最危险的失败不是漏掉某个点，而是"尽快收敛出一个看起来完整的交付物"——
表格齐、阶段全、有自我批判，看着像回事，深度却是空的。广度的体面最会骗人。
本 SOP 全部机制，只为对治这一件事。

## 根本命题

渗透的本质：把「攻击者能否在授权范围内达成危害 X」这一命题，从**未知**压缩为
**已证实为真（带 PoC）** 或 **已证伪（带证据）**。

一个漏洞假设只有三段生命：未知 → 疑似 → 确认 / 证伪。
- 只有 **确认（带 PoC）** 与 **证伪（带证据）** 是终态，是成果。
- 中间态（识别、罗列、单点验证）全是**库存**，不是产物。

机制的目的不是"防 AI 糊弄"（那是监工思维），而是"定义什么叫真正做完"。
这是目标驱动的标准，对人对机一视同仁。

## 四条根本准则（全部规则的推导起点）

> 下面所有具体规则（五项强制 / 五跳 / 攻击面穷举 / 谎报治理 / 验收清单）都是
> 这四条准则的推论，不是并列堆叠的清单项。改规则前先问：它是哪条准则的推论？

**准则一 · 证据决定状态**（治 F2 谎报）
任何关于系统的断言，其状态只由**可被独立复现的证据**决定。无证据 = 无断言；
证据不足 = 最高只能标"疑似·待复核"，不计入评级。证据必须是原文（原始请求 +
响应 + 复现步骤 + 判定依据），非复述的"我看到"。反射 / 回显 / 报错只是信号，不是证据。
**OOB（Out-of-Band）类证据**（DNS/HTTP 回连日志、可复现的时间差阈值）与请求/响应
原文**同等有效**，不因"没有直接响应"被误判为无证据——盲 SSRF / 盲 XXE / 盲命令注入 /
时间盲注等无法直接观测响应的漏洞类，走此证据路径确认，不因证据形式特殊被强降"疑似"。

**准则二 · 终态决定完成**（治 F1 半成品当交付）
工作的最小单元 = 一个假设推到终态（利用成功，或用证据证伪）。识别 / 罗列 / 单点
验证均为中间态，不得写进结论、不算"已测完"。结构完整 ≠ 做完；只要有一个假设卡
在中间态，任务即未完成。火力必须集中：宁可 P0 打穿 3 个，不要 30 个接口各打一层皮。

**准则三 · 攻击面决定假设集**（治 F3 漏假设 · F5 用例脱离语义）
假设集必须从系统功能 / 攻击面推导，而非套固定清单。凡"有哪些框架 / 角度 / 弱点
类"由执行者穷举列出，含冷门的，不等人喂。每个漏洞类标 适用 / 不适用 + 理由；
某类未被列出的唯一合法理由是"攻击面中无触发它的功能"，绝不能是"清单里没写"。
**payload / FUZZ 字典 / 候选路径本身也是假设**，同受本准则约束：必须能回答
"为什么对它用这条"，通用清单只作兜底补充，不得作为主力。

**准则三推论 · 科学推理测试模型（治 F5 的元框架）**
> 本推论的提出源于 face-recognition-api.djicorp.com 项目的实战验证。
> 核心认知：**不要从"攻击清单"出发，要从"理解系统"出发。**
>
> 这不是一个技巧，这是一个**元思维方式**——
> 把"我该试什么攻击？"替换成"这个系统是什么、怎么建的、哪里会坏？"
> 适用于所有测试场景，不只是 403 绕过。

**模型定义**：面对任何安全测试任务，先回答五个问题，再动手。不回答完这五个问题 = 测试还未开始：

**Q1：这是什么系统？（身份识别）**
- 不是猜名字，是**定位它在技术栈中的层**：
  - 这是 WAF？API 网关？应用框架？自定义中间件？
  - 每层的防御原语完全不同，绕过策略必须由层决定
- 识别信号来自响应头（`server`、`X-Powered-By`、`X-AspNet-*`）、响应体格式（统一错误页 vs 框架报错 vs 原始 HTTP）、行为模式（统一 403 vs 路径差异 vs 方法差异）

**Q2：如果我是开发者，我会怎么构建它？（共情建模）**
- 技术选型：语言、框架、协议、数据库、中间件——带选择理由
- 业务链：完整的输入→处理→存储→输出→消费链路
- 敏感点：数据、凭证、管理面、审批流在哪
- 开发者最容易在哪犯错：默认配置没关、调试头没删、鉴权没全覆盖、版本没更新

**Q3：这个系统最可能在哪里失效？（假设生成）**
- 由 Q1 的层和 Q2 的技术选型，推出 3-5 个针对性假设
- 每个假设必须能回答"为什么这个系统会在这里出问题"——答案来自 Q1 和 Q2
- 禁止回答"因为常见漏洞清单上有这一项"

**Q4：我怎么验证这个假设？（实验设计）**
- 用**变量隔离法**：控制组（已知通过的请求）→ 只改一个变量 → 观察响应变化
- 变量维度：方法/路径/Content-Type/请求头/协议/请求体/认证头
- 每个实验记录：控制组响应 → 实验组响应 → 推断

**Q5：响应告诉我什么？下一步怎么走？（循环迭代）**
- 每次响应都是一个信号——即使是 403/401 也有信息量
- 如果假设被证伪：不是"这个系统没漏洞"，而是"我的理解模型需要修正"
- 切维度继续：路径不通切 CT，CT 不通切方法，方法不通切协议
- 直到规则被完全画出，或者所有维度都穷举完毕

**完整推导关系**：
```
Q1（身份识别）→ 我知道在和什么系统打交道
    ↓
Q2（共情建模）→ 我知道它是怎么建成的，容易在哪里出错
    ↓
Q3（假设生成）→ 我推出它最可能在哪些点失效
    ↓
Q4（实验设计）→ 我设计实验来验证这些假设
    ↓
Q5（循环迭代）→ 我从每个响应中学习，修正模型，继续验证
    ↓
    ↗ 直到规则被画出（发现入口）或所有维度穷举（证伪）
```

### 推论的具体实例：开发者共情模型作为 Q2

> 开发者共情模型是 **Q2（共情建模）** 在上述元框架中的具体实现。

在完成 Q1（身份识别）之后，必须强制切换角色——**先当开发者，再当测试者**。
这个"切换"不是心理暗示，而是必须输出**结构化产物**的硬约束：

**① 业务身份建模（回答"我在开发什么"）**
- 系统类型 / 行业定位 / 核心功能
- 如果是你开发这个系统，技术选型会怎么做（语言、框架、协议、数据库、中间件）
- 这个系统的业务链是什么（输入→处理→存储→输出→消费）
- 敏感操作和关键数据在哪

**② 开发者常见失误清单（回答"我容易在哪里犯错"）**
- 框架默认行为：Actuator / Swagger / H2 Console / DevTools 是否忘记关
- 协议选择陷阱：REST vs gRPC vs GraphQL，暴露了哪些预期外的接口
- 鉴权实现盲区：全局拦截器是否覆盖所有方法？调试头是否忘记删？
- 配置泄露：默认密钥、调试模式、环境标识是否残留在生产环境
- 组件版本：哪些依赖存在已知 CVE？是否需要升级？

**③ 从开发者视角生成假设（回答"我会怎么攻击自己写的系统"）**
- 如果我是这个开发者，我最担心哪个接口被暴露？
- 我可能会忘记保护哪个端点？
- 我可能会在生产环境遗留什么调试入口？
- 攻击者最可能从哪里切入？

> 以上三步输出**必须落在纸面上**（写入业务语义模型表的"系统类型判定"和
> "针对性假设"列），才算完成 Q2。不输出 = 没做。
> 这个模型是 Q2 的高频实例，不是替代 Q1-Q5 元框架。**先做 Q1（身份识别），
> 再做 Q2（共情建模），然后走 Q3-Q5 循环。**

**准则四 · 对称校准**（治 F4 评级偏差）
漏报与谎报是对称的失败，同等严重。评级 = 真实可利用性，不为显高危拔高，不为显
干净压低。拿不准就降级标疑似——这不是保守，是校准，不许赌。

## LLM 五种根本失效（验收时对照"有没有复发"）

| 编号 | 失效 | 本质 | 对治准则 |
| --- | --- | --- | --- |
| F1 | 停在中间态当终态交付（识别即完成、铺得好看） | 优化"看起来完整"而非"钻到底" | 准则二 |
| F2 | 无充分证据即断言终态（谎报） | 优化"显得有料"而非"是真的" | 准则一 |
| F3 | 假设集不完整（漏攻击面、等人喂） | 优化"答已知"而非"穷举未知" | 准则三 |
| F4 | 误差不对称（为显高危拔高 / 为显干净藏报） | 优化"显得专业"而非"校准" | 准则四 |
| F5 | 用例脱离语义（通用 payload 硬套、火力平摊、打不中重点） | 优化"跑过了流程"而非"打中了这个系统" | 准则三（深化） |

> F5 是准则三的深化——payload / 字典 / 测试用例本身也是"假设"，同样必须从攻击
> 面语义推导，不套清单、不平摊。全部具体规则均由上述准则推导，非并列堆叠。

## 协作范式

- **AI 是知识源，你是审计**：别喂框架清单，让它自己识别 + 穷举；你不懂的它补。
- **激活式提问**：用"穷举/所有/还有什么/专家会但新手漏"逼它倒空知识。
- **触发器只给判据，不给案例**：新增识别类规则时，只写"什么信号下该警觉"，
  不写"命中后有哪些具体手法"——具体手法模型自己已掌握，写出来是对着已经会
  的人重复教一遍，白占上下文；真正稀缺、需要人编辑判断力标定的只是"识别信号"
  这一层。已验证的具体历史模式（如 CVE 引用）例外，那是证据溯源规范（治
  F2 谎报），不是知识灌输，按 HF-1/HF-4②现状继续保留全文引用。

## 五项强制必做（准则三的高频实例，是下限不是上限）

> 这五项不是独立清单，而是准则三"攻击面决定假设集"每次运行都会命中的高频结果，
> 只覆盖技术指纹层，全做完也≠做完。高危常出在它们没列举的维度（尤其业务逻辑），
> 那条线由下方「攻击面穷举」兜底。把跑完五项当收工，正是准则二要打击的 F1。

1. **指纹必转 CVE**：每个组件——语言/Web 框架（Spring Boot、Shiro、Spring
   Security、Python/Go 各框架）、中间件、前端库——必须**搜索 CVE + 逐条验证可
   利用性**，输出编号 + 适用性 + 验证结论。只识别版本不查 CVE = 未完成。
   > 前提：执行端需可联网查 NVD/CVE 库；不能联网时必须**主动声明局限**，
   > 标注"基于训练知识，需人工用 NVD 复核"，禁止假装查过。
   > 反幻觉硬规：CVE 编号必须**可溯源**（给 NVD/厂商公告链接，或标"未核实·
   > 基于训练知识"）；适用性必须**落到目标实测特征**（响应头/报错/行为指纹），
   > 禁止仅凭版本号比对就下"可利用"。宁可标"疑似·待核实"，不许自信报错号。
2. **JS 必须全量啃完**：所有 JS（含压缩/混淆大包）360° 无死角——全部字符串
   字面量 + request/axios 调用点 + router 路由表 + 内联配置/密钥/token。
   要求只有一个：**挖全**。禁单正则、禁以耗 token 跳过。产物：补全接口×参数矩阵。
   目标无 Web SPA、以 APP 为主时，同等要求转施于 APK 静态提取（反编译后啃字符串
   字面量/接口调用点/路由表/密钥），不因"这是 APP 不是网页"而降低挖掘标准。
3. **漏洞探测必 FUZZ**：对 JS 挖出的每个敏感接口/参数必须 FUZZ——注入
   （SQLi/命令/模板/路径穿越）、边界（超长/空/特殊编码）、参数污染、
   type confusion、越权。负结论必带观测证据，禁单点验证判死。
4. **权限绕过必做，按四步递进序列执行**：接口未授权 → 接口权限绕过 → 水平越权 →
   垂直越权，逐步推进，不许跳步或只测一步就收工。
   > 前提：输入必须是 HF-2（JS/APK 挖掘）+ HF-3.1（路径 FUZZ）产出的**全量接口
   > 清单**，不在残缺清单上测——清单不全，越权测试的假设集本身就是错的。
   - **① 接口未授权**：去 token/session 直接调，判断该接口是否本就无需鉴权。
   - **② 接口权限绕过**：JWT 操纵（alg=none、RS256→HS256、弱密钥、kid 注入、
     不校验 exp/aud/iss）+ 鉴权头绕过（XFF、X-Original-URL、X-Rewrite-URL、
     Referer、调试/内部头）+ 路径/方法绕过。
**框架身份驱动绕过（Q1-Q5 应用实例）**：遵循 Q1→Q2→Q3→Q4 流程——
      Q1（识别鉴权框架身份：Spring Security / Shiro / 自定义）→
      Q2（构建该框架已知绕过模式的模型）→
      Q3（推导本系统最可能存在的绕过点）→
      Q4（设计实验验证）。绕过手法必须由 HF-1 指纹识别出的
      鉴权框架身份决定，再查该框架已知的路径匹配/大小写/编码差异类绕过模式，
     禁止不看框架身份直接套通用清单——通用清单命中率低，是清单思维不是攻击面
     思维。高频实例锚点（**非穷尽**，遇到其他框架同样先识别身份再查其已知模式）：
     - Java + Shiro：路径 normalization 差异（`;`分号插入、`//`、`%2e%2e/`、
       大小写不敏感），代表性历史模式参考 CVE-2020-1957（Servlet 容器与 Shiro
       路径解析不一致）——**仅作模式参考，不代表目标必然命中，命中与否仍需
       实测验证**。
     - Java + Spring Security：Filter 层 pattern 与路由层解析不一致（trailing
       slash、大小写、`;jsessionid=` 插入），代表性历史模式参考 CVE-2022-22978
       （RegexRequestMatcher 大小写问题）——同样仅作模式参考。
     - PHP：类型混淆（`==` 弱比较/magic hash）、参数污染（PHP 取最后一个重复
       参数，与其他语言行为不同）。
   - **③ 水平越权**：同权限层级账号间的数据越权。
   - **④ 垂直越权**：跨权限层级的数据/操作越权。
     **若获得 ≥2 个不同权限账号，③④以账号间交叉对比为主，不以单账号猜测
     ID 为主**——A 账号 token 操作 B 账号数据、低权 token 试高权接口、
     token/session 互换、同一接口不同角色响应的字段级差异也算信号。零/单账号
     场景才退回未授权访问尝试与 ID/资源标识符枚举。
   **错误驱动参数回填（Q4 实验设计的应用）**：①②步绕过/未授权访问成功但
   接口报错缺参数时，**优先用响应报错信息回填参数名**；报错信息不足才转第 3
   项的参数 FUZZ，不倒序——报错里明摆着的参数名不去读、直接上字典，是舍近求远。
5. **资产暴露面必测**：对象存储（阿里云 OSS / MinIO 等）与常见未授权中间件——
   桶权限（公共读/公共写、ListObjects 是否放开）、AK/SK 是否硬编码泄露（回链
   JS/APK 挖掘产物）、STS 临时凭证越权换取、presigned URL 篡改扩权/延期、MinIO
   默认凭证与匿名策略配置错误；兜底覆盖未授权 Redis/Elasticsearch/MongoDB、
   CI 面板、`.git`/`.env` 目录暴露。只识别"用了 OSS/MinIO"不测权限配置 = 未完成。

## 两条正交轴

纵轴(做什么) × 横轴(怎么和 AI 干)，交叉 = 一个动作。给人理解用；
给 AI 时拍平成 L2 线性指令，AI 看不到"轴"。

纵轴 8 阶段：`0 授权 → 1 侦察 → 2 建模 → 3 发现 → 4 验证 → 5 关联 → 6 交付 → 7 复测`
横轴 4 方法：激活式提问 / 五跳推导链 / 结论闸 / 人审计·AI供给

阶段闸：每阶段先出结构化产物且**无半成品残留**，才进下一步。

## 五跳推导链（组件无关）

```
识别 → 穷举弱点类(标适用/不适用+理由) → 假设 → 执行(推到利用或证伪) → 关联
```
走到跳4 才算这个组件做完。你审跳1·2，AI 发挥跳3·4。

## 攻击面穷举（治"清单≠覆盖"）

五项强制是固定动作，覆盖不了应用专属的洞。覆盖必须**从功能而非组件出发**穷举：
对每个业务功能，列出适用的漏洞类——含业务逻辑层：越权(水平/垂直)、金额/数量
篡改、状态机绕过、竞态、SSRF、反序列化、二阶注入、OAuth/SSO 流程缺陷、密码
重置/找回逻辑、缓存投毒；**含实时协议层**：WebSocket（连接鉴权/消息层鉴权/
Origin 校验）、MQTT（匿名连接/topic 订阅发布权限/client_id 伪造）、自定义信令
（如 drtc 类，需协议逆向）——每类标**适用/不适用+理由**，适用的走五跳到跳4。

准则三的推论：覆盖来自对攻击面的穷举，不来自任何固定清单。固定清单只兜底，不封顶。

### 识别信号加注（准则三的推论：只给判据，不给案例）

业务逻辑层清单里以下四类，命中判据容易被漏判，专门给识别信号（不给具体案例，
命中后由执行者自行调用已掌握技术；真实项目中出现的具体案例回灌 L3，不进本节）：

- **多入口防护不一致**：同一业务逻辑（登录/验证码/找回密码/支付确认等）是否
  存在多个可达入口（Web/APP/旧版接口/合作方接口/内部接口）？逐一列出入口，
  核对限流/验证码/锁定策略是否在每个入口都一致生效——只测其中一个入口就下
  "已限流"结论 = 未完成。
- **状态机中间态被当终态**（准则二"终态才算完成"反向应用于目标系统自身的
  安全逻辑）：目标的鉴权/注册/审批/支付等流程是否为多步骤？逐步标注每一步
  产生的凭证/session/权限，是否提前具备了最终步骤才该有的能力。
- **长效凭证签发强度弱于短效凭证**：目标是否同时存在长效凭证（remember-me
  token、密码重置 token、长期 API 密钥）与短效凭证（session、一次性验证码）？
  核对两者绑定强度（设备指纹/一次性使用/有效期/是否可枚举）是否对称，还是
  长效凭证反而校验更松。
- **委托身份断言校验缺陷**：目标是否存在委托身份验证（OAuth/SAML/JWT 联合
  登录）？核对断言/token 完整性校验（签名算法锁定、签发方/受众校验、一次性
  使用、状态防重放）是否完整，而非只看"能否登录成功"。

## 危害升级（准则二的推论：终态不只是"触发"，还要把危害证明到位）

停在"触发了"就交付，是准则二要打击的另一种 F1——漏洞类型对了、复现了，但危害
证明不到位，交付价值大打折扣。每类高频漏洞升级到什么程度才算终态：

- **SSRF** → 尝试打云元数据接口（阿里云 `100.100.100.200`，不要默认套 AWS 的
  `169.254.169.254`）、内网管理面，而非停在"响应时间变长/能探测端口"。
- **IDOR** → 证明"可批量枚举"（脚本化跑通一批 ID），而非改一个 ID 看一条记录。
- **RCE** → PoC 给到非破坏性验证（`whoami`/`id`/读一个非敏感文件）即算终态，
  不做进一步破坏，但不能停在"命令注入点存在"这一步。
- **反射型信号（XSS/CSRF 等）** → 证明能否链到会话劫持/后台操作，而非停在弹窗。

这不要求把每个洞都打到最坏结果，而是要求**明确回答"危害升级到哪一步、为什么停在这一步"**，
不许在"刚触发"这一步就默认收工。

## 谎报治理（确认必带可复现 PoC）

漏报的对称失败是**谎报**（F4，见上表）：把 200 脑补成 SQLi、把反射当 XSS 不验
执行、为显高危夸大评级、报无源 CVE。一份充满幻觉高危的交付，危害不亚于一份没深度的。

准则一/四的推论：**任何"确认漏洞"必须带可独立复现的 PoC**（原始请求+响应、复现
步骤、判定依据），否则只能记"**疑似·待复核**"，不得写进确认结论、不计入危害评级。

- 证据是**可独立验证的原文**（请求/响应原貌、时间戳），不是 AI 复述的"我看到"。
- 反射/回显/报错只是**疑似信号**，未验到实际执行/读取/越权前，不算确认。
- 评级按**真实可利用性**，拿不准就降级标疑似，不许为显高危而赌。
- 与"证伪带证据"对称：负结论带覆盖证据，正结论带复现证据，两侧都不许空口。
