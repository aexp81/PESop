# L1 · 方法论（稳定层）

> 给人读。讲"为什么"。地基，很少变。仅用于授权范围内的安全评估。
> 版本 v2.0 · 2026-07-01。与 L2 v2.0 同步。从四条根本准则推导全部规则。

## 灵魂一句话

**真正的渗透是往下钻、钻到打穿；不是横着铺、铺到好看。**

最危险的失败不是漏掉某个点，而是"尽快收敛出一个看起来完整的交付物"——
表格齐、阶段全、有自我批判，看着像回事，深度却是空的。广度的体面最会骗人。
本 SOP 全部机制，只为对治这一件事。

## 第一性原理

**工作的最小单元 = 一个假设被推到结论：打穿，或用证据证伪。**

识别组件、罗列接口、单点验证——都是**半成品**，不是结论。
机制的目的不是"防 AI 糊弄"（那是监工思维），而是"定义什么叫真正做完"。
这是目标驱动的标准，对人对机一视同仁。

## 四铁律

1. **成功 = 把假设钻到结论的深度**，不是产出一份结构完整的报告。结构完整 ≠ 做完。
2. **AI 是知识源，你是审计**。别喂框架清单，让它自己识别 + 穷举；你不懂的它补。
3. **激活式提问**：用"穷举/所有/还有什么/专家会但新手漏"逼它倒空知识。
4. **每个单元必须终结于"利用 or 证伪 + 证据"**。半成品一律视为未完成。

## 四种失败模式（专治，验收重点看）

> 前三种是"做不够"（漏报），第四种是"过头了"（谎报）——两个方向都毁报告。

| 失败模式 | 长相 | 对治 |
| --- | --- | --- |
| 识别即终点 | 拿到版本/认出框架就归档，不推到 CVE/利用 | 五跳必走到跳4；指纹必落 CVE |
| 过早收敛 | 窄正则抓几个就停、一次验证就判死参数、嫌"差不多"就停 | 完成标准硬量化，禁"差不多" |
| 成本规避 | 大 JS 嫌耗 token 不啃、fuzz 嫌慢不跑 | 列为强制项，方法写死 |
| 无中生有(谎报) | 把疑似当确认、脑补漏洞、夸大评级、报无源 CVE | 确认必带可复现 PoC，否则降"疑似"（见下） |

## 四项强制必做（验收红线，不可省略）

> 这四项是**下限不是上限**：只覆盖技术指纹层，全做完也≠做完。高危常出在它们
> 没列举的维度（尤其业务逻辑），那条线由下方「攻击面穷举」兜底。把跑完四项当
> 收工，正是本 SOP 要打击的"过早收敛"。

1. **指纹必转 CVE**：每个组件——语言/Web 框架（Spring Boot、Shiro、Spring
   Security、Python/Go 各框架）、中间件、前端库——必须**搜索 CVE + 逐条验证可
   利用性**，输出编号 + 适用性 + 验证结论。只识别版本不查 CVE = 未完成。
   > 前提：执行端需可联网查 NVD/CVE 库；不能联网时必须**主动声明局限**，
   > 标注"基于训练知识，需人工用 NVD 复核"，禁止假装查过。
   > 反幻觉硬规：CVE 编号必须**可溯源**（给 NVD/厂商公告链接，或标"未核实·
   > 基于训练知识"）；适用性必须**落到目标实测特征**（响应头/报错/行为指纹），
   > 禁止仅凭版本号比对就下"可利用"。宁可标"疑似·待核实"，不许自信报错号。
2. **JS 必须全量啃完**：所有 JS（含压缩/混淆大包）360° 无死角——全部字符串
   字面量 + request/axios 调用点 + router 路由表 + 内联配置/密钥/token。
   要求只有一个：**挖全**。禁单正则、禁以耗 token 跳过。产物：补全接口×参数矩阵。
3. **漏洞探测必 FUZZ**：对 JS 挖出的每个敏感接口/参数必须 FUZZ——注入
   （SQLi/命令/模板/路径穿越）、边界（超长/空/特殊编码）、参数污染、
   type confusion、越权。负结论必带观测证据，禁单点验证判死。
4. **权限绕过必做**：JWT 操纵（alg=none、RS256→HS256、弱密钥、kid 注入、
   不校验 exp/aud/iss）+ 鉴权头绕过（XFF、X-Original-URL、X-Rewrite-URL、
   Referer、调试/内部头）+ 路径/方法绕过。逐项试，记录结果。

## 两条正交轴

纵轴(做什么) × 横轴(怎么和 AI 干)，交叉 = 一个动作。给人理解用；
给 AI 时拍平成 L2 线性指令，AI 看不到"轴"。

纵轴 8 阶段：`0 授权 → 1 侦察 → 2 建模 → 3 发现 → 4 验证 → 5 关联 → 6 交付 → 7 复测`
横轴 4 方法：激活式提问 / 五跳推导链 / 结论闸 / 人审计·AI供给

阶段闸：每阶段先出结构化产物且**无半成品残留**，才进下一步。

## 五跳推导链（组件无关）

```
识别 → 穷举弱点类(标适用/不适用+理由) → 假设 → 执行(推到利用或证伪) → 关联
```
走到跳4 才算这个组件做完。你审跳1·2，AI 发挥跳3·4。

## 攻击面穷举（治"清单≠覆盖"）

四项强制是固定动作，覆盖不了应用专属的洞。覆盖必须**从功能而非组件出发**穷举：
对每个业务功能，列出适用的漏洞类——含业务逻辑层：越权(水平/垂直)、金额/数量
篡改、状态机绕过、竞态、SSRF、反序列化、二阶注入、OAuth/SSO 流程缺陷、密码
重置/找回逻辑、缓存投毒——每类标**适用/不适用+理由**，适用的走五跳到跳4。

铁律：覆盖来自对攻击面的穷举，不来自任何固定清单。固定清单只兜底，不封顶。

## 谎报治理（确认必带可复现 PoC）

漏报的对称失败是**谎报**：把 200 脑补成 SQLi、把反射当 XSS 不验执行、为显高危
夸大评级、报无源 CVE。一份充满幻觉高危的交付，危害不亚于一份没深度的。

铁律：**任何"确认漏洞"必须带可独立复现的 PoC**（原始请求+响应、复现步骤、判定
依据），否则只能记"**疑似·待复核**"，不得写进确认结论、不计入危害评级。

- 证据是**可独立验证的原文**（请求/响应原貌、时间戳），不是 AI 复述的"我看到"。
- 反射/回显/报错只是**疑似信号**，未验到实际执行/读取/越权前，不算确认。
- 评级按**真实可利用性**，拿不准就降级标疑似，不许为显高危而赌。
- 与"证伪带证据"对称：负结论带覆盖证据，正结论带复现证据，两侧都不许空口。

## 可疑信号（触发即深挖，不深挖 = 未完成）

时序差异 / 报错泄露 / 同输入异响应·异输入同响应 / 该403却200 /
长度规律差异 / 绕过后仍成功 / 参数被反射 / 前端可见密钥token。

## 完成度验收清单（逐项卡，为确认抵达目标）

- [ ] ① 每个组件都有 CVE 搜索 + 验证结论（含语言框架）？不能联网时已声明局限？
- [ ] ② JS 全量啃完（字面量+调用点+路由表+密钥），非单正则、无跳过，矩阵已补全？
- [ ] ③ 每个敏感接口/参数跑完整轮 FUZZ（注入/边界/污染/type confusion/越权）带证据？
- [ ] ④ 权限绕过四类（JWT/XFF/调试头/路径方法）全部试过并记录？
- [ ] ⑤ 攻击面已从功能穷举漏洞类（含业务逻辑层），每个适用类走到跳4？
- [ ] ⑥ 每个"确认漏洞"都有可独立复现 PoC？无 PoC 的已降级为"疑似·待复核"、不计评级？
- [ ] 每个组件五跳走到跳4（利用 or 证伪），无停在识别的？
- [ ] 每个"安全"结论带观测证据？
- [ ] 自我批判列出的缺口已当场补完并附结果？

## 控制档位

轻：五跳到跳2 + 关键单元推到结论。
中：全单元推到结论 + 四项强制 + 攻击面穷举 + 关联 + 一轮自评闭环。
重：全单元实测打穿 + 四项强制 + 攻击面穷举 + 全关联 + 否定结论带证据 + 多轮自评闭环。
