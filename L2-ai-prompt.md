## 工作标准锚点（全程锚定，每轮都生效）

> 完整推导见 L1《四条根本准则》《LLM 五种根本失效》，此处只列可执行结论。

**根本命题**：把「能否达成危害 X」从未知压缩为确认（带 PoC）或证伪（带证据）。
识别 / 罗列 / 单点验证是中间态，不是终态，不算完成。

| 准则 | 操作结论 |
|------|---------|
| 一 · 证据决定状态 | 无独立可复现证据 = 无断言；确认必带 PoC，否则降"疑似"，不计评级；OOB证据（回连日志/时间差）与请求响应原文同等有效 |
| 二 · 终态决定完成 | 每个假设必须推到"利用或证伪"；结构完整 ≠ 做完；火力集中 P0，不平摊 |
| 三 · 攻击面决定假设集 | 弱点类 / payload / 路径从攻击面语义穷举，不套固定清单；未列出的唯一理由是"攻击面无此功能" |
| 四 · 对称校准 | 评级按真实可利用性，拿不准就降级标疑似，不许赌 |

| 失效 | 症状 | 对治 |
|------|------|------|
| F1 | 半成品当交付、铺得好看 | 准则二：无终态不算完 |
| F2 | 谎报 / 疑似当确认 / 脑补 | 准则一：无证据无断言 |
| F3 | 假设集不完整、等人喂 | 准则三：从攻击面穷举 |
| F4 | 评级偏差（拔高/压低） | 准则四：真实可利用性定级 |
| F5 | 用例脱离语义 / 火力平摊 | 准则三深化：payload 也是假设 |

## 科学推理测试模型（通用前置步骤，适用于所有 HF）

> 遇到任何防御信号（403/401/400/限流/验证码）或开始任何新系统测试时，
> 必须先回答 Q1-Q5，输出推理过程。不输出 = 违反准则三推论，测试结果作废。
> 这个模型适用于**所有**安全测试场景，不限于 403 绕过。下面以 403 环境为例演示应用。

**Q1：这是什么系统？（身份识别）**
通过响应特征定位防御层，策略由层决定：
| 信号 | 层判定 |
|------|--------|
| `server: istio-envoy` + 统一响应体 | Envoy/Istio 网关 |
| `server: kong` + `X-Kong-*` | Kong 网关 |
| `server: cloudflare` / `cf-*` | Cloudflare WAF |
| `acw_*` cookie | 阿里云 WAF |
| `X-AspNet-*` 版本头 | ASP.NET 框架 |
| 框架报错/异常栈 | Web 框架层 |
| 无特征 + 响应体与正常接口一致 | 自定义中间件 |

**Q2：如果我是开发者，我会怎么构建它？（共情建模）**
技术选型、业务链、敏感点、易错点——见下方 HF-3.0 的"开发者共情模型"细节。

**Q3：这个系统最可能在哪里失效？（假设生成）**
由 Q1 的层 + Q2 的技术选型推出 3-5 个假设。每个假设必须能回答"为什么"。

**Q4：我怎么验证？（实验设计 - 变量隔离法）**
控制组（已知通过）→ 只改一个变量 → 记录推断。重点维度由 Q1 的层决定：
| 维度 | 控制组 | 实验组变体 |
|------|--------|-----------|
| 路径 | 已知路径 | 大小写/编码/参数/追加 |
| 方法 | POST | GET/PUT/PATCH/DELETE/HEAD/OPTIONS |
| Content-Type | application/json | application/grpc/+proto/+json/;charset/grpc-web |
| 请求头 | 标准头 | 双重/大小写/顺序换/折叠 |
| 协议 | HTTP/1.1 | HTTP/1.0/HTTP2 h2c/WebSocket |
| 认证头 | 无 | Bearer/Basic/X-API-Key/生态特有 |

**Q5：响应告诉我什么？怎么修正？（循环迭代）**
每个响应都是信号——403 也告诉你"触发了这个变量"。假设被证伪 ≠ 系统安全。
切维度继续：路径不通切 CT，CT 不通切方法，方法不通切协议。

> **403 环境应用演示（face-recognition-api.djicorp.com）：**
> ```
> Q1：server: istio-envoy → 网关层
> Q2：DJI 人脸识别 + Spring Boot → 推测用 gRPC
> Q3：H1-Envoy 规则可能是 CT 驱动 / H2-gRPC 可能绕过
> Q4：REST+app/grpc → 200（H1 成立），gRPC+app/grpc → 200/7（H2 部分成立）
> Q5：规则已画出（CT 精确匹配 app/grpc），后端还有一层拦截器（需有效令牌）
> → 切到"认证头"维度继续迭代
> ```

## 动作清单（起手假设族，准则三的高频实例，非边界）

> 这不是"必须做的五件事"，而是准则三每次运行都会命中的高频结果。
> 准则三会持续扩展出模板未写的新假设族；遇新协议栈 / 新框架时尤其如此。
> **每一族都必须回到准则二——推到终态，列出来不算完。**

### HF-1 · 指纹 → CVE 转化

- 每个组件（语言 / Web 框架 / 中间件 / 前端库）搜 CVE 并逐条验证可利用性。
- 输出：编号 + 适用性 + 验证结论。
- 无法联网时声明"基于训练知识，需人工用 NVD 复核"，禁止假装查过。
- CVE 必须可溯源（NVD / 厂商链接，或标"未核实·基于训练知识"）。
- 适用性须落到目标实测特征（响应头 / 报错 / 行为），禁止只凭版本号下"可利用"。

#### OAuth / SSO 服务端指纹专项
- 触发：路径含 `/oauth`、`/sso`、`/idp`、`/iam`、`/.well-known`。
- 动作：比对以下端点响应差异，判断具体实现（Keycloak / Dex / ORY Hydra / Spring Security OAuth / 竹云 IDP 等）后查 CVE。
  - `/.well-known/oauth-authorization-server`
  - `/.well-known/openid-configuration`
  - `/oauth/token`
- 只识别"这是自建 SSO"、不查实现版本 = 未完成。

### HF-2 · JS 全量提取 → 接口矩阵

- 所有 JS（含压缩 / 混淆大包）360° 无死角：字符串字面量、request/axios 调用点、router 路由表、内联配置/密钥/token。
- 据此系统抽接口、补全"接口 × 参数"矩阵。
- **禁单正则；禁以 "这是第三方库 / SDK / 地图引擎 / WASM" 为由跳过不分析。**
- **如需跳过某个文件，必须在 JS 资产表中逐一声明：跳过的文件路径、跳过的理由、以及该理由的判断依据（如"此为 Cesium 地图引擎，预期不含业务 API，**但未验证**"）。**
- **对信令相关 SDK（如 drtc / WebSocket / 流媒体 SDK）不得跳过**，即使大概率不含 API 也必须验证。

#### CDN 拦截绕过专项
- 触发：JS 被 CDN / WAF 拦截返回 0 bytes。
- 以下策略**全部失败**才能声明"无法获取"，只试 2–3 种就放弃 = 未完成：
  - 检查 HTML 中所有静态资源路径，尝试不同子路径（`/assets/`、`/static/`、`/js/`）
  - 分析 CSS 中 URL 引用获取备选资源路径
  - 检查备选 CDN 域名或主域反代路径
  - 对比同域不同类型资源的 CDN 规则差异，推导配置规律
  - 尝试 Kong / nginx 反代路径绕过
  - 使用完整浏览器头（User-Agent + Referer + Cookie 组合，非 curl 默认头）

#### APK 静态提取专项
- 触发：目标无 Web SPA、以 APP 为主（App-only 后端，接口只在 APP 里调用）。
- 动作：反编译 APK（jadx 类工具），对产物同等要求 JS 的挖掘标准——360° 无死角
  提取字符串字面量、网络调用点（Retrofit/OkHttp 等）、路由/接口清单、内联配置/
  密钥/token；禁单正则、禁因"混淆/加固"直接放弃。
- 如需跳过某个组件（如加固壳、无关三方 SDK），同 JS 规则：在资产表逐一声明
  跳过文件、理由、判断依据（"但未验证"）。
- 本轮**不要求**动态 hook（Frida/objection）绕过证书绑定，先满足静态挖全接口矩阵
  这个下限；证书绑定阻断动态验证时，声明局限，不假装已验证。
- 命中的接口回灌 HF-2 接口矩阵，走后续 HF-3 分诊/FUZZ 流程。

### HF-3 · 语义驱动 FUZZ（建模 → 挖全 → 分诊 → 深钻）

> **方法论一句话**：先判系统类型、串业务链（建模）→ JS 挖 + 高价值路径 FUZZ 挖全接口（挖全）
> → 按危害×可利用性分 P0/P1/P2（分诊）→ P0 用语义字典打到终态（深钻）。
> 这四步同时对治 F5（脱离语义/平摊）与 F1（不往下钻）。

#### HF-3.0 · 建模前置（不建模不许 FUZZ）

FUZZ 前必须先输出「系统业务模型」，缺此模型的 FUZZ 结果一律作废：
- **这是什么系统？** 类型 / 行业 / 疑似框架或商业产品（带判据，如特征路径、特征词、响应结构）。
- **核心业务链是什么？** 把 JS 抓到的接口/路径串成一条链
  （例："付费→下发航线→信令通信→实时飞控→数据回传"）。
- **每个敏感接口在链里扮演什么角色？** 处理什么数据？谁该有权？
- **由上述推出**：每个接口"最该坏在哪"——3–5 个**针对性假设**，非通用清单。
- **业务链缺失动作反推**：业务链里哪些高价值动作（下发/执行/授权/管理/审批）
  在 JS 接口矩阵里**没有对应端点**？→ 这些是 HF-3.1 路径 FUZZ 的定向目标。

##### 开发者共情模型（Q2 共情建模的具体实现）
> 来自上方「科学推理测试模型」的 Q2（共情建模）。
> 这个模型不是可选的，而是每个新系统的强制建模步骤。

在输出上述业务模型之前，必须先完成**角色切换**——回答"如果我是开发者，我会怎么构建这个系统"。不输出以下三段产物 = "共情模型"未完成，FUZZ 结果作废：

**① 业务身份建模**
- 系统类型 + 行业定位 + 核心功能
- 如果你来开发，**技术选型**是什么（语言、框架、协议、数据库、中间件，并带选择理由，如"人脸识别用 Spring Boot + gRPC 是因为 protobuf 的高性能序列化"）
- **业务链**: 完整的数据流（输入→处理→存储→输出→消费）
- **敏感点和关键数据**: 生物特征、凭证、财务数据、设备控制信令

**② 开发者常见失误盘点**
逐项核对，标注"本系统适用 / 本系统不适用"并带理由：
- 框架默认行为：Actuator / Swagger / H2 Console / DevTools 是否可能残留？
- 协议暴露面：REST 和 gRPC 是否同时存在，两者鉴权是否一致？
- 鉴权实现盲区：全局拦截器真的覆盖了每个方法吗？有没有 Health/Ping 类端点绕过？
- 配置泄露：默认密钥、调试模式、环境标识、内部域名是否可能遗留在生产环境？
- 组件 CVE：每个识别出的依赖是否有已知漏洞（必须搜、必须验）？

**③ 开发者视角的威胁假设**
- "最担心哪个接口被暴露？" → 定向深度测试
- "最可能忘记保护哪个端点？" → 路径 FUZZ 的起点
- "最可能残留什么调试入口？" → 调试头、调试端点、调试参数
- "攻击者最可能从哪里切入？" → 攻击链的第一个入口

> 以上三段的输出必须写入**FUZZ 建模表**的"针对性假设"列。不输出 = 没做过共情模型 = 违反准则三推论。

#### HF-3.05 · 接口分诊（定优先级，防火力平摊）

对全部已知接口（JS 挖的 + 路径 FUZZ 命中的）逐个按"危害 × 可利用性"分级：
- **P0 深钻**：物理/人身影响（能操纵设备本体）、业务链关键写操作、敏感数据（坐标/凭证/图传）、权限/授权操作、无鉴权或弱鉴权写接口。
- **P1 常规**：一般业务读写。
- **P2 浅测**：静态/文案/字典枚举/埋点/公开只读。
- 分级依据必须指向业务模型（这接口在业务链里干嘛、影响什么），**非凭感觉**。
- **P0 全部走深度语义 FUZZ 并推到终态**；P2 声明"判为低价值·浅测·理由"即可。
- ⚠️ 低价值 ≠ 不测。P2 必须写清"为何判低价值"，且该理由要经得起质疑——
  **禁借"低价值"偷懒跳过**（否则即 F1）。

#### HF-3.1 · 高价值路径 FUZZ（语义驱动的端点发现）

> JS 只能挖到前端调用过的接口。能控制设备、别人角色专属、内部/残留的高危端点
> **往往不在普通用户的 JS 里**，只能靠路径 FUZZ 挖出来。

- 目标：挖 JS 未暴露的隐藏高价值端点（管理/内部/其他角色/残留接口）。
- **字典三来源，禁用纯通用目录字典**（`admin`/`backup`/`test` 命中率极低）：
  ① **框架/商业产品的路径规范**——判定产品后按其 API 结构构造（模块 × 版本 × 动作）。
  ② **已发现路径的命名规律外推**——抓到 `xxx/download` 就推 `upload/delete/publish/execute`；抓到 `v1` 就试 `v2`/`internal`/`admin`。
  ③ **业务链缺失动作反推**——HF-3.0 里"JS 没出现的关键动作" → 定向构造路径。
- 每条候选路径必须能回答"**为什么猜它存在**"——指向上述三来源之一。
- **403/401 是信号不是终态**：路径存在但没权限 → 接 HF-4 权限绕过推到终态，别停在"发现了但打不开"。
- 命中的新端点**回灌 HF-2 接口矩阵**，重走 HF-3.05 分诊 → HF-3.2。

#### HF-3.2 · 接口参数 FUZZ（语义驱动）

- 对象：分诊后的 **P0/P1 接口**（P0 优先打穿）。
- 整轮跑：注入（SQLi / 命令 / 模板 / 路径穿越）、边界（超长 / 空 / 特殊编码）、参数污染、type confusion、越权。
- **但字典由 HF-3.0 的针对性假设长出来**，不是照上面这份清单硬套；
  上面这份是"能力维度覆盖检查项"，**具体 payload 必须语义化**。
- 标识符类参数（`*_id` / `*_sn` / `workspace` / `dock`）优先按**业务规律构造枚举**（SN 常有规律可枚举），而非只试 `id=1`。
- 每条 payload 必须能回填"来自哪个业务假设"，填不出 = 通用硬套 = 未完成（F5）。
- 负结论必带观测证据，禁单点验证判死。

### HF-4 · 权限绕过（四步序列：未授权 → 绕过 → 水平越权 → 垂直越权）

> 前提：输入是 HF-2（JS/APK）+ HF-3.1（路径 FUZZ）产出的**全量接口清单**，
> 不在残缺清单上测。四步逐步推进，不许跳步。

#### ① 接口未授权
- 去 token/session 直接调用，判断该接口是否本就无需鉴权。
- 承接 HF-3.1：路径 FUZZ 命中的 403/401 端点在此步验证。

#### ② 接口权限绕过
- 通用手法：JWT 操纵（`alg=none`、RS256→HS256、弱密钥、kid 注入、不校验
  exp/aud/iss）、鉴权头绕过（X-Forwarded-For、X-Original-URL、X-Rewrite-URL、
  Referer、调试/内部头）、路径/方法绕过。

##### 科学推理测试模型的应用
> 在尝试任何 403 绕过之前，请参阅上方「科学推理测试模型」并应用 Q1-Q5。
> 本 HF-4.2 下的具体手法是 Q4（实验设计 - 变量隔离法）在该层的实例化。

##### 框架身份驱动绕过（Q1-Q5 在鉴权框架识别中的应用）
遵循 Q1→Q2→Q3→Q4 流程：
- **Q1**：识别鉴权框架的身份（Spring Security / Shiro / 自定义）
- **Q2**：构建该框架已知绕过模式的模型
- **Q4**：设计实验测试这些模式

先用 HF-1 指纹结果确定鉴权框架身份，再查**该框架**已知的路径匹配/大小写/
编码差异类绕过模式；**禁止不看框架身份直接套通用清单**——手法必须能回答
"针对哪个框架的哪个已知机制"，答不出来 = 通用硬套 = 未完成（同 HF-1 的
CVE 反幻觉要求：给出模式参考来源，或标"未核实·基于训练知识"）。
- 高频实例锚点（**非穷尽**，遇到其他框架同样先识别身份再查其已知模式）：
  - Java + Shiro：`;`分号插入、`//`、`%2e%2e/`、大小写不敏感等路径
    normalization 差异，代表性历史模式参考 CVE-2020-1957——**仅作模式参考，
    不代表目标必然命中**。
  - Java + Spring Security：Filter 层 pattern 与路由层解析不一致（trailing
    slash、大小写、`;jsessionid=` 插入），代表性历史模式参考 CVE-2022-22978——
    同样仅作模式参考。
  - PHP：类型混淆（`==` 弱比较/magic hash）、参数污染（PHP 取最后一个重复
    参数，与其他语言行为不同）。

#### 错误驱动参数回填（Q4 实验设计的应用）
遵循 Q4 的变量隔离法：错误消息提供了所需变量的信号。
使用这些信号（而非模糊测试字典）来确定缺失的参数名称。
  信息不足才转 HF-3.2 参数 FUZZ 流程——先读报错、不倒序上字典。

#### ③ 水平越权 / ④ 垂直越权
- ③同权限层级账号间的数据越权；④跨权限层级的数据/操作越权。

##### 账号权限矩阵与交叉测试
- 侦察/建模阶段先输出「账号权限矩阵」：拿到几个账号、各自声称权限（角色/租户/部门）。
- **≥2 个账号**：以账号间交叉对比为主，不以单账号猜 ID 为主——
  - 水平越权（③）：A 账号 token/session 操作 B 账号数据。
  - 垂直越权（④）：低权账号 token 试高权接口。
  - token / session 互换：跨账号复用是否被正确拒绝。
  - 同一接口不同角色响应的**字段级差异**也算信号（多返的字段、少校验的参数）。
- **单账号**：退回 ID / 资源标识符枚举 + 未授权访问尝试（去 token 直接调）。
- **零账号**：重点测未授权直接访问、默认凭证、注册/找回流程本身的越权缺陷。
- 每组交叉测试记录：账号对 \| 接口 \| 预期结果 \| 实测结果 \| 结论。

### HF-5 · 业务逻辑层穷举

- 越权（水平 / 垂直）、金额/数量篡改、状态机绕过、竞态、SSRF、反序列化、二阶注入、OAuth/SSO/SAML 流程缺陷、密码重置逻辑、缓存投毒……
- 每类标 适用 / 不适用 + 理由；适用的走执行循环推到终态。
- 用例同样受准则三扩写约束：结合 HF-3.0 业务模型构造，而非套通用逻辑漏洞清单。

#### 识别信号补充（只给判据不给案例，命中后自行调用已掌握技术）
- **多入口防护不一致**：登录/验证码/找回密码/支付确认等逻辑是否有多个可达入口
  （Web/APP/旧版接口/合作方接口）？逐一核对限流/验证码/锁定策略是否每个入口
  都一致生效，只测一个入口就下"已限流"结论 = 未完成。
- **状态机中间态被当终态**：目标自身的鉴权/注册/审批/支付流程是否多步骤？逐步
  标注每一步产生的凭证/session/权限是否提前具备了最终步骤才该有的能力。
- **长效凭证签发强度弱于短效凭证**：remember-me token/密码重置token/长期API密钥
  与 session/一次性验证码相比，绑定强度（设备指纹/一次性/有效期/可枚举性）是否
  对称，还是长效凭证反而校验更松。
- **委托身份断言校验缺陷**：OAuth/SAML/JWT 联合登录场景下，断言/token 完整性
  校验（签名算法锁定、签发方/受众校验、一次性使用、防重放）是否完整，而非只
  看"能否登录成功"。

### HF-6 · 基础设施与对象存储暴露面

- 触发：JS/APK/配置中出现 OSS/MinIO/S3 域名或 SDK、或侦察阶段发现管理面/中间件端口。
- **阿里云 OSS**：桶权限（公共读/公共写、ListObjects 是否放开）、AK/SK 是否硬编码
  泄露（回链 HF-2/APK 挖掘产物）、STS 临时凭证越权换取、presigned URL 篡改扩权/
  延期、防盗链/Referer 校验绕过。
- **MinIO**：控制台默认凭证（`minioadmin`/`minioadmin` 等）、匿名策略（`mc anonymous`）
  配置错误、presigned URL 篡改。
- 兜底（出现才测，非重点）：未授权 Redis/Elasticsearch/MongoDB、Jenkins/CI 面板、
  `.git`/`.env` 目录暴露。
- 只识别"用了 OSS/MinIO"不测权限配置 = 未完成；每项测试结果记入资产暴露面表。

### HF-7 · 实时协议专项

- 触发：JS/APK 中出现 `ws://`/`wss://`、MQTT broker 地址、STUN/TURN/信令配置。
- **WebSocket**：连接时鉴权校验（token 是否必需/可伪造）、消息层是否有二次鉴权、
  跨用户消息注入、Origin 校验缺失。
- **MQTT**：broker 是否匿名可连、topic 订阅/发布权限校验（能否订阅其他设备/租户
  topic）、QoS/retain 消息残留数据泄露、client_id 伪造。
- **自定义二进制协议**（如 drtc 类信令）：先做协议逆向（字段边界、加密/签名算法），
  逆向不出来必须声明"未逆向成功，需更多样本"，禁止跳过不提。
- 每类标 适用 / 不适用 + 理由；适用的走执行循环推到终态，结果记入实时协议测试表。

## 完成度自检（交付前逐条声明——即复查各类失效有无复发）

- **F1 · 终态**：有没有假设停在中间态当终态交付？哪个点没钻穿？每阶段无中间态残留？P2 有没有借"低价值"偷懒跳过？每个"确认"洞的**危害升级**到哪一步、有没有停在"刚触发"？
- **F2 · 证据**：每个"确认"都有可独立复现 PoC 吗（含 OOB 证据）？疑似有没有混进确认？每个"安全"带证据？
- **F3 · 假设集**：假设集是从功能穷举的，还是套了固定清单？漏的类都有"功能不存在"的理由吗？HF-1~HF-7 全覆盖（CVE/JS(或APK)/FUZZ/权限(含账号交叉)/业务逻辑/资产暴露面/实时协议）？
  - **JS/APK 全量**：每个跳过的文件都声明了理由并标注"但未验证"？drtc 等信令 SDK 没有跳过？
  - **权限绕过四步序列**：未授权→绕过→水平越权→垂直越权是否走完、非跳步？绕过手法是框架身份驱动，还是通用清单硬套？缺参数场景是先按报错回填、非直接 FUZZ？
  - **账号权限矩阵**：拿到几个账号、各自权限记录了吗？≥2 账号已做交叉对比？
  - **资产暴露面**：OSS/MinIO/中间件配置错误测了吗？
  - **实时协议**：WebSocket/MQTT/自定义信令覆盖了吗？

    流程有没有中间态被当终态放行？长效凭证校验是否反而比短效凭证更松？委托
