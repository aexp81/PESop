# PESop 骨架决策记录（ADR）

> 本文件记录 PESop 长期迭代的**地基架构共识**：不是实现清单，而是"为什么这么定"。
> 后续任何迭代/重构，都应先对照本文件——**改扩展点，不改原生骨架**。
> 状态：v0.1（骨架定盘）· 定盘依据见文末「决策溯源」。

---

## 0. 两条不可动摇的原则（一切决策的裁决标尺）

任何设计在纳入本项目前，必须同时通过这两条：

1. **原则一 · 避免过度设计**：engine（代码层）只做"确定性、机器该做"的事；凡属"思路、判断、发挥"的，一律留给 AI，不塞进代码。
2. **原则二 · 极强扩展性**：新增能力（新域、新攻击面、新下限、新 FUZZ 策略）应通过**追加扩展点**实现，**不改动原生骨架**。骨架是稳定的底盘，能力是可插拔的。

> 冲突裁决：当"想做某个功能"与这两条原则冲突时，原则优先，功能让路或降级为扩展点。

## 1. 项目的精髓：到底是什么（定盘结论）

经过多轮对齐，对"项目精髓"的最终表述如下（修正了"Q1-Q5 + FUZZ 就是精髓"的初始判断）：

### 1.1 Q1-Q5 与 FUZZ 是「发挥上限的双引擎」，不是骨架

- **Q1-Q5 是"想"**：把渗透从"套攻击清单"变成"理解系统→共情建模→生成针对性假设"的推理内核。它是 AI 突破下限、往高价值漏洞发散的**思维引擎**。
- **FUZZ 是"做"**：执行假设的主力手段，遵循"接口→参数→业务"的深度递进，本质是 Q4（实验设计）里"变量隔离法"的落地。
- 两者是**服务于目的的手段**，不是目的本身。

**engine 对这两个引擎的正确姿态（原则一的直接推论）：**
- 对 Q1-Q5：**只托管产物 + 卡点**（`intel.model` 存建模；application 域没建模就判"覆盖下限未达标"）。engine **不实现** Q1-Q5 的推理。
- 对 FUZZ：**只托管发包存证**（`http_client` 已有）**+ 把"接口/参数/业务"递进作为覆盖下限检查项**。engine **不实现** FUZZ 策略，策略是 AI 的发挥。

### 1.2 真正的骨架是「下限守门层」（floor guard）

项目的**目的**是产出有价值的漏洞（严重/高危/中危），其次是不漏测、测完整。由此推出：

> 指挥这个项目的不是"技术信号（如响应差异）"，而是"**价值密度**"——投入产出比。
> engine 骨架的顶层，是一个**只守下限、不设上限**的守门员。

- **守下限**（防漏测、防偷懒、防跑偏）：该覆盖的攻击面没覆盖、现有信息没榨干，就不许判"走不通"、不许收工。
- **不设上限**（放开发挥）：下限之上，AI 可无限发散、深挖，engine 不拦、不封顶。

这一刀同时满足两条原则：engine 只做"检查下限"这件确定性的事（原则一），下限是可增删的检查项清单（原则二）。

## 2. 核心概念定义（钉死语义，防后续漂移）

### 2.1 「走不通」的定义（本项目最关键的语义）

**走不通 ≠ 技术失败（如返回 403、响应无差异）。走不通 = 价值未达成 且 努力已充分。**

形式化为一个**合取条件**：

```
走不通 =  价值未达成
          AND
          努力已充分（= 覆盖下限达标 AND 榨干下限达标）
```

- **价值未达成**：findings 里没有 confirmed 的**中危及以上**漏洞。
- **努力已充分**（两者都要，且是**下限**，不设上限）：
  - **覆盖下限**：该覆盖的攻击面都过了假设——三域（infra/framework/application）覆盖，application 域已建模，已知接口（intel.endpoints）都验证过。
  - **榨干下限**：现有信息顺完了——intel 里的 secrets/hosts/endpoints 没有"拿到却没用"的悬挂线索。

> 关键补充（用户定盘）：覆盖 + 榨干"两者都要，但这是**下限**，最低完成即可，**不设上限**，要充分发挥模型能力"。
> 即：下限未达标 → engine **禁止**判走不通（防过早发散/漏测）；下限达标 → 发不发散、挖多深，**AI 说了算**（engine 不封顶）。

### 2.2 「收敛」与「测完整」的关系（局部 vs 全局）

- **收敛是局部的**：某条攻击线挖到高危/严重 → 这条线价值已达成 → 可收敛（可浅收，不必在这条线继续无限发散）。
- **测完整是全局的**：单条线收敛**不豁免**全局下限——其他域/攻击面仍须覆盖，否则算漏测。

> 一句话：挖到高价值可以收敛**这条线**，但整体完整性（不漏测）要求其他攻击面仍要覆盖到下限。

### 2.3 「假设」与「变思路」的关系

- **产出假设是常态**：每一步推理都在产假设（Q1-Q5 持续运转）。
- **变思路是事件**：由"这条线走不通"（见 2.1）触发的特殊动作——切维度、换协议、换建模角度（对应 L1 的 Q5）。
- 二者不是一回事：顺着信息深入 ≠ 变思路；只有价值未达成且努力充分，才触发变思路。

## 3. 骨架分层（定盘架构图）

**不推翻现有四层结构**（它是对的），只在其上新增一个"下限守门"薄层，把双引擎与情报/证据串成闭环。

```
┌──────────────────────────────────────────────────────────────────────────┐
│  已有地基（不动）：                                                          │
│    http_client 发包存证 · intel 情报库(共享黑板) · evidence 价值账本(findings) │
└──────────────────────────────────────────────────────────────────────────┘
                                     ↑ 读态势(intel + findings)
┌──────────────── 新增薄层：下限守门 floor guard ────────────────────────────┐
│  唯一职责:回答"现在够不够收工 / 够不够判走不通"                             │
│                                                                            │
│  下限 = 两组【可扩展的检查项】(satisfied / not-yet):                        │
│    A. 覆盖下限:infra 覆盖? framework 覆盖? application 已建模+覆盖?          │
│                intel.endpoints 都验证过?                                    │
│    B. 榨干下限:intel.secrets/hosts/endpoints 有没有"拿到却没用"的悬挂线索?   │
│                                                                            │
│  输出【缺口清单】而非【下一步指令】:                                        │
│    下限未达标 → "禁止判走不通,还差:[具体缺口]"  ← 守下限、防漏测            │
│    下限达标   → "下限已满足,发散与否 / 挖多深由你定"  ← 不设上限            │
└────────────────────────────────────────────────────────────────────────────┘
        ↑ 决策权在 AI:发散/建模/变思路/怎么打,engine 只给缺口清单,不下指令
┌──────────────── Q1-Q5 建模 + FUZZ 执行（AI 做,上限空间）───────────────────┐
│  产出 → 写 findings(价值) + 写 intel(信息) → 回喂下限守门重新评估(闭环)      │
└────────────────────────────────────────────────────────────────────────────┘
```

## 4. 三个关键设计决策（每条都为两原则服务）

### 决策一：下限守门做成「检查项清单」，不做成「硬编码流程」
- 每条下限是独立、可插拔的检查（覆盖 infra？榨干 secrets？）。
- 新增下限（如"实时协议层 WebSocket 要覆盖"）= append 一条检查项，**骨架零改动**。
- 命中原则二（扩展性）。

### 决策二：engine 只输出「缺口清单」，不输出「下一步指令」
- 守门员说"你还差 infra 域没覆盖"（陈述缺口），**不说**"去打 Druid"（下达指令）。
- 缺口怎么补、用什么思路，是 AI 的上限空间。
- 命中原则一（不把 AI 的脑子塞进代码）+ 守住"engine 守下限、AI 发挥上限"的边界。

### 决策三：最大化复用现有 intel + findings，几乎不造新概念
- "覆盖下限"读 intel.fingerprints / endpoints（已有）。
- "榨干下限"读 intel.secrets / hosts（已有）+ 标记"用过没"。
- "价值达成"读 findings.severity / status（已有）。
- 唯一新增：一份极薄的"下限定义"（哪些算下限）+ 一个"读态势算缺口"的守门函数。
- 命中原则一（新增面极小，不过度设计）。

## 5. 双引擎在骨架中的最终归位

| 引擎 | 定位 | engine 只做 | engine 不做 |
| --- | --- | --- | --- |
| Q1-Q5 | 发挥上限的思维引擎 | 托管建模产物(intel.model) + 卡下限(没建模不许收工) | 不实现推理/不自动生成假设 |
| FUZZ | 发挥上限的执行引擎 | 托管发包存证 + 把"接口/参数/业务"递进作为覆盖下限检查项 | 不实现 FUZZ 策略/不自动跑字典 |

> FUZZ 的策略路径（由 AI 依信息量选择，engine 不干涉）：
> - 路径甲（有信息，如 JS 已提取接口）：业务语义 + 参数验证为主。
> - 路径乙（无信息，只有系统名）：Q2 开发者共情造假设 → 验证；TOP 字典枚举仅兜底。

## 6. 边界红线（防止后续迭代跑偏）

以下行为**违反骨架宪法**，后续任何迭代都不得引入：

1. ❌ 让 engine 自动判断"走没走通/该不该变思路"并替 AI 决策（越过原则一）。
2. ❌ 让 engine 自动生成假设、自动选 FUZZ 策略、自动跑字典（把 AI 的活代码化）。
3. ❌ 给"发挥"设上限（如限制发散次数、限制假设数量）——只守下限，不封顶。
4. ❌ 为新增能力去改动原生骨架，而不是追加扩展点（违反原则二）。
5. ❌ 守门员输出"下一步指令"而非"缺口清单"（越过 engine/AI 职责边界）。

## 7. 决策溯源（这些结论怎么来的）

| 决策 | 来源 |
| --- | --- |
| 精髓是"下限守门"而非"Q1-Q5/FUZZ 孤立点" | 用户明确"目的是产出高价值漏洞，其次不漏测"→ 推出价值密度是指挥棒 |
| "走不通"= 价值未达成 且 努力充分 | 用户纠正："走不通不是 403，是没信息也没挖到有价值漏洞" |
| 覆盖+榨干"两者都要，是下限，不设上限" | 用户定盘原话 |
| 收敛是局部、测完整是全局 | 用户确认同意 |
| 驱动型（轻驱动/状态机驱动），非纯文本 | 用户提出疑问 → 分析得出：每步依赖上步结构化结果，故须驱动；但决策权归 AI，故是轻驱动 |
| 复用 intel/findings、不造新系统 | 原则一（避免过度设计）的直接应用 |
